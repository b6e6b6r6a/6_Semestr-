**Требования к тестировщику:**
1. Теория тестирования и документация
2. Знания предмета который тестируется
3. Использовать инструменты автоматического тестирования

**Тестирование:**
1. Включен во все активности жизненного цикла
2. Статическое (без сполнения кода) и динамическое(во время выполнения кода)
3. Оценивается ожидание и реальность
4. Объект - программа(что сравнивается): отжельный модуль, компонент, подсистема, система ; базис - документация (с чем сравнивается)


**Дефект(То же самое что и баг):**
1. Программа не делает то, что она должна делать
2. Программа делает то, что не должна
3. Программа делает то, что не упоминалось
4. Программа неудобна

# ЛК2 12.02.2024 Разработка через тестирование TDD (test driven development)

Жизненный цикл ПО

1. Каскадная модель:

 1)под тестирование выделен только один этап, дефекты во время эксплуатации нельзя исправить
     
 2)часто нет плана тестирования

![image](https://github.com/b6e6b6r6a/6_Semestr-/assets/113089548/34b4d339-3d67-4ecf-8ac0-a2bd0bc3e2b6)

2. V-модель:

 1)тестируется каждый этап, но возврата нет

![image](https://github.com/b6e6b6r6a/6_Semestr-/assets/113089548/d16c48fa-c5b3-4b58-bbd3-1321dcfdc40a)

3. Итеративная или инкрементальная модель:

 1)большие объемы регрессионного тестирования
   
 2)отстутствие четкого плана для обнаружения и устранения дефектов

4. Спиральная модель:
   
  1)сложно тестировать на ранних этапах, но тестирование становится ключевым этапом

  Уровень тестирования:

   а) компонентное - каждую функцию тестируем отдельно

   в) интеграционное - проверили что например из базы удалятся данные

   с) системное - все вместе

 ![image](https://github.com/b6e6b6r6a/6_Semestr-/assets/113089548/23c15093-aa56-47fe-a64b-90710ae247eb)

  2)приемочное

    Тестирование бывает:

 а) сатическое - без исполнения кода

 в) динамическое - с исполнением кода

![image](https://github.com/b6e6b6r6a/6_Semestr-/assets/113089548/c86a464a-04d8-4107-ac01-7f2af53decda)

Тестирование Яндекс Диска:

1. функциональные тесты :
   - загрузки файлов
   - синхронизации данных
   - обмен файлами
2. тесты производительности:
   - скорость загрузки файлов
   - синхронизация данных между устройствами
3. нагрузочные тесты:
   - одновременная загрузка файлов
   - синхронизация и скачивание файлов
4. тестирование совместимости:
   - работа на различных устройствах
5. позитивные тесты:
   - правильная организация файлов и папок - файлы и папки должны успешно создаваться, перемещаться, переименовываться и удаляться без потери данных
   - загрузка файлов разных размеров - все загруженные файлы должны успешно появиться на Яндекс.Диске и быть доступны для использования
6. негативные тесты:
   - не авторизованные пользователь не может загружать файлы на диск
   - Яндекс.Диск должен останавливать загрузку файла, превышающего доступное место на диске, и уведомлять пользователя о превышении лимита
7. исследовательские тесты:
   - устойчивости работы при одновременной нагрузке
8. модульное тестирование:
   - тестирование каждого раздела
   - тестирование объема диска
9. интеграционное тестирование:
    - авторизоваться на Яндекс.Диск с использованием учетных данных Яндекса
    - синхронизировать файлы или данные между Яндекс.Диском и Яндекс.Облаком
10. системное тестирование:
    - нельзя получить доступ к файлам без авторизации
    - проверить передачу файлов и данных между различными сервисами

# ЛК3 19.02.2024 Дефекты

**Отчет о дефекте** - документ, содержащий отчет о любом недостатке в компоненте или системе, который может привести компонент или систему к невозможности выполнить требуемую функцию.

![image](https://github.com/b6e6b6r6a/6_Semestr-/assets/113089548/3f236c79-9fe2-4325-8245-76a196f737d1)

Отчет о дефекте:
1. идентификатор
2. краткое описание: характер проблемы, границы дефекта, полседствие дефекта
3. подробное описание: ожидаеые и фактические результаты
4. влияние: критичность и приоритет

Причины почему дефекты не устраняют:
1. нехватка времени
2. дефект вовсе не дефект
3. устранять неисправность слишком рискованно
4. это того не стоит
5. неэффективный отчет о дефектах

**Упражнение 1:** Заголовок: Некорректные результаты сложения в калькуляторе Windows.

Описание: При проведении тестирования калькулятора в операционной системе Windows были обнаружены некорректные результаты при выполнении сложения четных чисел

Шаги вопроизведения:

Запустить калькулятор Windows.
Выполнить операцию сложения четных чисел.
Посмотреть результат сложения.
Ожидаемый результат: Калькулятор должен корректно выполнять арифметические операции и возвращать правильные результаты сложения.

Наблюдаемый результат: Калькулятор выдает неправильные результаты сложения для четных чисел.

Влияние: Критичное.

**Тестовый сценарий**

Тестовый набор:
1. Тестовые сценарии
2. Тестовые данные
3. Правила генераци

Выбор данных:
1. Покрытие операторов
2. Узлов ветвления  

**Классы эквивалентности** - одинаковая вероятнось обнаружения конкретного типа ошибок
